# Spec: Add Agent-Generated Graphs Display Panel

## Purpose
Add a dedicated panel that can render graphs generated by the agent from structured data returned during chat interactions.

## Problem Statement
Agent responses that include analytical data are currently displayed as plain text, making trends and comparisons harder to understand quickly.

## Goals
- Provide a panel for visualizing chart-ready data produced by the agent.
- Support multiple chart types for common analysis use cases.
- Keep graph rendering synchronized with the active chat/session context.
- Gracefully handle invalid or missing graph payloads.

## Non-Goals
- No advanced BI/dashboard builder in this feature.
- No user-authored chart scripting language.
- No export-to-PDF in this initial version.

## User Stories
- As a user, I can view graphs generated by the agent in a dedicated panel.
- As a user, I can switch between graph outputs associated with a chat.
- As a user, I can understand when a graph could not be rendered and why.

## Functional Requirements
1. App must include a graph panel adjacent to or below chat content (responsive by viewport).
2. Agent/backend must be able to return structured graph payloads in chat responses.
3. Frontend must parse and validate graph payload schema before rendering.
4. Panel must support at least chart types:
   - `line`
   - `bar`
   - `pie`
5. If multiple graphs are returned, user can navigate/select among them.
6. Graphs must be tied to the active chat; switching chats updates graph panel context.
7. If no graph data exists for current chat, show a helpful empty state.

## UX Requirements
- Panel title: "Graphs" or "Analysis".
- Include compact metadata for each graph:
  - title
  - chart type
  - optional source message timestamp
- Include loading state while graph payload is being processed.
- Include error state with readable message when payload invalid.
- Allow collapsing/hiding panel to prioritize chat area on small screens.

## Data Contract Requirements
### Graph Payload Envelope (from backend/agent)
```json
{
  "type": "graph",
  "graph": {
    "id": "graph_001",
    "title": "Weekly Revenue",
    "chartType": "line",
    "xLabel": "Week",
    "yLabel": "Revenue",
    "series": [
      {
        "name": "Revenue",
        "data": [
          { "x": "W1", "y": 1200 },
          { "x": "W2", "y": 1500 }
        ]
      }
    ],
    "options": {
      "stacked": false,
      "showLegend": true
    }
  }
}
```

### Validation Rules
- `graph.id`: non-empty string.
- `graph.chartType`: one of supported types.
- `series`: non-empty array.
- each data point must include x and y values compatible with chart type.

## Backend Requirements
1. Extend chat response contract to include optional graph payload(s).
2. Ensure graph payloads are structured JSON, not markdown-only chart descriptions.
3. Return consistent schema with explicit `type: graph` discriminator.
4. Include structured error details if graph generation fails.

### Chat Response Example (with graph)
```json
{
  "success": true,
  "data": {
    "message": "Here is the trend chart.",
    "artifacts": [
      {
        "type": "graph",
        "graph": {}
      }
    ]
  }
}
```

## Frontend Requirements
1. Add graph artifact support in chat/message types.
2. Add graph state in store:
   - `graphsByChatId`
   - `selectedGraphIdByChatId`
3. Add graph panel component with:
   - graph list/tabs selector
   - chart canvas/render area
   - empty/loading/error states
4. Integrate chart library (for example: Chart.js, ECharts, or similar) with typed adapter layer.
5. Render graphs only from validated schema; reject and report invalid payloads.
6. Preserve graph selection per chat when user switches chats.

## Performance Requirements
- Avoid full panel re-render when unrelated chat messages arrive.
- Memoize transformed chart datasets where practical.
- Cap maximum points per series for safe rendering or downsample if payload is very large.

## Accessibility Requirements
- Provide textual summary fallback for each graph (title + key values/trends).
- Ensure keyboard navigation for graph selector list.
- Ensure color palette has sufficient contrast and is colorblind-friendly where possible.

## Edge Cases
1. Unsupported chart type:
   - show "Unsupported chart type" message and keep raw summary available.
2. Malformed data points:
   - skip invalid points when safe; otherwise mark graph as invalid.
3. Very large dataset:
   - apply point limit/downsampling and show note to user.
4. Chat switch during render:
   - cancel stale render/update and show active chat graphs only.

## Security and Validation
- Treat all graph payloads as untrusted input.
- Validate schema before rendering.
- Escape/sanitize any labels/titles shown in DOM.
- Enforce payload size limits on backend response.

## Acceptance Criteria
1. Graph panel appears in UI and is responsive across desktop/mobile layouts.
2. When agent returns valid graph payload, chart renders correctly.
3. Multiple graphs can be selected/viewed for the active chat.
4. Switching chats updates panel to that chat's graphs.
5. Invalid payloads do not crash UI and produce clear error feedback.
6. Existing chat text flow remains functional with no regressions.

## Test Cases
- Unit:
  - Graph payload validator accepts valid schema and rejects invalid schema.
  - Store maps graph artifacts by chat ID correctly.
- Component:
  - Graph panel empty/loading/error states.
  - Chart selector updates rendered chart.
- Integration:
  - Send prompt that triggers graph artifact and verify chart appears.
  - Switch between chats with/without graphs and verify panel updates.
  - Malformed graph payload returns safe error rendering.
- Performance:
  - Render stress test with large series payload within acceptable frame budget.

## Implementation Notes
- Keep graph payload contract versionable (e.g., optional `schemaVersion`).
- Consider starting with one chart library and an adapter interface for future swaps.
- Keep graph extraction logic close to message parsing layer to avoid duplication.
